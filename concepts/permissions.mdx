Claude Code는 에이전트가 수행할 수 있는 작업과 수행할 수 없는 작업을 정확하게 지정할 수 있는 세분화된 권한을 지원합니다. 권한 설정은 버전 관리에 체크인하여 조직의 모든 개발자에게 배포할 수 있으며, 개별 개발자가 맞춤 설정할 수도 있습니다.

## 권한 시스템

Claude Code는 강력함과 안전성의 균형을 맞추기 위해 계층화된 권한 시스템을 사용합니다:

| 도구 유형 | 예시 | 승인 필요 여부 | "네, 다시 묻지 마세요" 동작 |
| :--- | :--- | :--- | :--- |
| 읽기 전용 | 파일 읽기, Grep | 아니오 | 해당 없음 |
| Bash 명령 | 셸 실행 | 예 | 프로젝트 디렉토리 및 명령별로 영구 저장 |
| 파일 수정 | 파일 편집/쓰기 | 예 | 세션 종료 시까지 |

## 권한 관리

`/permissions` 명령으로 Claude Code의 도구 권한을 확인하고 관리할 수 있습니다. 이 UI에는 모든 권한 규칙과 해당 규칙이 정의된 settings.json 파일이 표시됩니다.

* **Allow** 규칙은 수동 승인 없이 지정된 도구를 사용할 수 있게 합니다.
* **Ask** 규칙은 Claude Code가 지정된 도구를 사용하려 할 때마다 확인을 요청합니다.
* **Deny** 규칙은 Claude Code가 지정된 도구를 사용하지 못하게 차단합니다.

규칙은 **deny -> ask -> allow** 순서로 평가됩니다. 첫 번째로 일치하는 규칙이 적용되므로, deny 규칙이 항상 우선합니다.

## 권한 모드

Claude Code는 도구 승인 방식을 제어하는 여러 권한 모드를 지원합니다. 각 모드의 사용 시기에 대해서는 [권한 모드](/en/permission-modes)를 참조하세요. [설정 파일](/en/settings#settings-files)에서 `defaultMode`를 설정합니다:

| 모드 | 설명 |
| :--- | :--- |
| `default` | 표준 동작: 각 도구의 첫 사용 시 권한을 요청합니다 |
| `acceptEdits` | 세션 동안 파일 편집 권한을 자동으로 수락합니다 |
| `plan` | Plan 모드: Claude가 분석할 수 있지만 파일 수정이나 명령 실행은 불가합니다 |
| `auto` | 백그라운드 안전 검사를 통해 도구 호출을 자동 승인하며, 요청에 부합하는 작업인지 확인합니다. 현재 연구 프리뷰 단계입니다 |
| `dontAsk` | `/permissions` 또는 `permissions.allow` 규칙으로 사전 승인된 도구를 제외하고 자동 거부합니다 |
| `bypassPermissions` | 보호 디렉토리 쓰기를 제외한 권한 프롬프트를 건너뜁니다 (아래 경고 참조) |

<Warning>
  `bypassPermissions` 모드는 권한 프롬프트를 건너뜁니다. `.git`, `.claude`, `.vscode`, `.idea` 디렉토리에 대한 쓰기는 리포지토리 상태 및 로컬 설정의 우발적 손상을 방지하기 위해 여전히 확인을 요청합니다. `.claude/commands`, `.claude/agents`, `.claude/skills`에 대한 쓰기는 Claude가 스킬, 서브에이전트, 명령을 생성할 때 일상적으로 사용하므로 예외이며 프롬프트가 표시되지 않습니다. 이 모드는 Claude Code가 손상을 일으킬 수 없는 컨테이너나 VM 같은 격리된 환경에서만 사용하세요. 관리자는 [관리형 설정](#관리형-설정)에서 `disableBypassPermissionsMode`를 `"disable"`로 설정하여 이 모드를 차단할 수 있습니다.
</Warning>

`bypassPermissions` 또는 `auto` 모드의 사용을 방지하려면 [설정 파일](/en/settings#settings-files)에서 `permissions.disableBypassPermissionsMode` 또는 `disableAutoMode`를 `"disable"`로 설정하세요. 이 설정은 재정의할 수 없는 [관리형 설정](#관리형-설정)에서 사용할 때 가장 유용합니다.

## 권한 규칙 구문

권한 규칙은 `Tool` 또는 `Tool(specifier)` 형식을 따릅니다.

### 도구의 모든 사용 매칭

도구의 모든 사용을 매칭하려면 괄호 없이 도구 이름만 사용합니다:

| 규칙 | 효과 |
| :--- | :--- |
| `Bash` | 모든 Bash 명령을 매칭 |
| `WebFetch` | 모든 웹 페치 요청을 매칭 |
| `Read` | 모든 파일 읽기를 매칭 |

`Bash(*)`는 `Bash`와 동일하며 모든 Bash 명령을 매칭합니다.

### 세분화된 제어를 위한 지정자 사용

괄호 안에 지정자를 추가하여 특정 도구 사용을 매칭합니다:

| 규칙 | 효과 |
| :--- | :--- |
| `Bash(npm run build)` | 정확히 `npm run build` 명령을 매칭 |
| `Read(./.env)` | 현재 디렉토리의 `.env` 파일 읽기를 매칭 |
| `WebFetch(domain:example.com)` | example.com에 대한 페치 요청을 매칭 |

### 와일드카드 패턴

Bash 규칙은 `*`를 사용한 glob 패턴을 지원합니다. 와일드카드는 명령의 어느 위치에나 나타날 수 있습니다. 이 설정은 npm 및 git commit 명령을 허용하면서 git push는 차단합니다:

```json  theme={null}
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Bash(git * main)",
      "Bash(* --version)",
      "Bash(* --help *)"
    ],
    "deny": [
      "Bash(git push *)"
    ]
  }
}
```

`*` 앞의 공백이 중요합니다: `Bash(ls *)`는 `ls -la`와 매칭되지만 `lsof`와는 매칭되지 않으며, `Bash(ls*)`는 둘 다 매칭됩니다. 레거시 `:*` 접미사 구문은 ` *`와 동일하지만 더 이상 사용되지 않습니다(deprecated).

## 도구별 권한 규칙

### Bash

Bash 권한 규칙은 `*`를 사용한 와일드카드 매칭을 지원합니다. 와일드카드는 시작, 중간, 끝 등 명령의 어느 위치에나 나타날 수 있습니다:

* `Bash(npm run build)` — 정확히 `npm run build` Bash 명령을 매칭
* `Bash(npm run test *)` — `npm run test`로 시작하는 Bash 명령을 매칭
* `Bash(npm *)` — `npm `으로 시작하는 모든 명령을 매칭
* `Bash(* install)` — ` install`로 끝나는 모든 명령을 매칭
* `Bash(git * main)` — `git checkout main`, `git merge main` 같은 명령을 매칭

끝에 공백 뒤에 `*`가 있으면(예: `Bash(ls *)`), 단어 경계를 강제하여 접두사 뒤에 공백이나 문자열 끝이 와야 합니다. 예를 들어 `Bash(ls *)`는 `ls -la`와 매칭되지만 `lsof`와는 매칭되지 않습니다. 반면 공백 없는 `Bash(ls*)`는 단어 경계 제약이 없으므로 `ls -la`와 `lsof` 모두 매칭됩니다.

<Tip>
  Claude Code는 셸 연산자(`&&` 등)를 인식하므로, `Bash(safe-cmd *)` 같은 접두사 매칭 규칙으로는 `safe-cmd && other-cmd` 명령을 실행할 수 있는 권한이 부여되지 않습니다.
</Tip>

"네, 다시 묻지 마세요"로 복합 명령을 승인하면, Claude Code는 전체 복합 문자열에 대한 단일 규칙이 아니라 승인이 필요한 각 하위 명령에 대해 별도의 규칙을 저장합니다. 예를 들어 `git status && npm test`를 승인하면 `npm test`에 대한 규칙이 저장되어, `&&` 앞에 무엇이 오든 이후의 `npm test` 호출이 인식됩니다. 하위 디렉토리로의 `cd` 같은 하위 명령은 해당 경로에 대한 자체 Read 규칙을 생성합니다. 단일 복합 명령에 대해 최대 5개의 규칙이 저장될 수 있습니다.

<Warning>
  명령 인수를 제한하려는 Bash 권한 패턴은 취약합니다. 예를 들어 `Bash(curl http://github.com/ *)`는 curl을 GitHub URL로 제한하려는 의도이지만, 다음과 같은 변형과는 매칭되지 않습니다:

  * URL 앞 옵션: `curl -X GET http://github.com/...`
  * 다른 프로토콜: `curl https://github.com/...`
  * 리다이렉트: `curl -L http://bit.ly/xyz` (github으로 리다이렉트)
  * 변수: `URL=http://github.com && curl $URL`
  * 추가 공백: `curl  http://github.com`

  보다 안정적인 URL 필터링을 위해 다음을 고려하세요:

  * **Bash 네트워크 도구 제한**: deny 규칙으로 `curl`, `wget` 및 유사 명령을 차단한 후, `WebFetch(domain:github.com)` 권한으로 허용된 도메인에 대해 WebFetch 도구를 사용
  * **PreToolUse 훅 사용**: Bash 명령의 URL을 검증하고 허용되지 않은 도메인을 차단하는 훅 구현
  * CLAUDE.md를 통해 허용된 curl 패턴에 대해 Claude Code에 지시

  WebFetch만 사용하는 것으로는 네트워크 접근을 방지할 수 없다는 점에 유의하세요. Bash가 허용되면 Claude는 여전히 `curl`, `wget` 또는 다른 도구를 사용하여 어떤 URL이든 접근할 수 있습니다.
</Warning>

### Read와 Edit

`Edit` 규칙은 파일을 편집하는 모든 내장 도구에 적용됩니다. Claude는 Grep, Glob 등 파일을 읽는 모든 내장 도구에 `Read` 규칙을 최선의 노력으로 적용합니다.

<Warning>
  Read와 Edit deny 규칙은 Claude의 내장 파일 도구에 적용되며, Bash 하위 프로세스에는 적용되지 않습니다. `Read(./.env)` deny 규칙은 Read 도구를 차단하지만 Bash에서의 `cat .env`는 차단하지 않습니다. 모든 프로세스의 경로 접근을 차단하는 OS 수준 강제를 위해서는 [샌드박스를 활성화](/en/sandboxing)하세요.
</Warning>

Read와 Edit 규칙은 모두 네 가지 고유 패턴 유형을 가진 [gitignore](https://git-scm.com/docs/gitignore) 사양을 따릅니다:

| 패턴 | 의미 | 예시 | 매칭 대상 |
| --- | --- | --- | --- |
| `//path` | 파일시스템 루트부터의 **절대** 경로 | `Read(//Users/alice/secrets/**)` | `/Users/alice/secrets/**` |
| `~/path` | **홈** 디렉토리부터의 경로 | `Read(~/Documents/*.pdf)` | `/Users/alice/Documents/*.pdf` |
| `/path` | **프로젝트 루트 기준** 상대 경로 | `Edit(/src/**/*.ts)` | `<프로젝트 루트>/src/**/*.ts` |
| `path` 또는 `./path` | **현재 디렉토리 기준** 상대 경로 | `Read(*.env)` | `<현재 디렉토리>/*.env` |

<Warning>
  `/Users/alice/file` 같은 패턴은 절대 경로가 아닙니다. 프로젝트 루트 기준 상대 경로입니다. 절대 경로에는 `//Users/alice/file`을 사용하세요.
</Warning>

Windows에서는 매칭 전에 경로가 POSIX 형식으로 정규화됩니다. `C:\Users\alice`는 `/c/Users/alice`가 되므로, 해당 드라이브 전체에서 `.env` 파일을 매칭하려면 `//c/**/.env`를 사용합니다. 모든 드라이브에서 매칭하려면 `//**/.env`를 사용합니다.

예시:

* `Edit(/docs/**)`: `<프로젝트>/docs/`의 편집 (NOT `/docs/`, NOT `<프로젝트>/.claude/docs/`)
* `Read(~/.zshrc)`: 홈 디렉토리의 `.zshrc` 읽기
* `Edit(//tmp/scratch.txt)`: 절대 경로 `/tmp/scratch.txt` 편집
* `Read(src/**)`: `<현재 디렉토리>/src/`에서 읽기

<Note>
  gitignore 패턴에서 `*`는 단일 디렉토리 내의 파일을 매칭하고, `**`는 디렉토리를 재귀적으로 매칭합니다. 모든 파일 접근을 허용하려면 괄호 없이 도구 이름만 사용하세요: `Read`, `Edit`, 또는 `Write`.
</Note>

### WebFetch

* `WebFetch(domain:example.com)` — example.com에 대한 페치 요청을 매칭

### MCP

* `mcp__puppeteer` — `puppeteer` 서버(Claude Code에서 설정된 이름)가 제공하는 모든 도구를 매칭
* `mcp__puppeteer__*` — `puppeteer` 서버의 모든 도구를 매칭하는 와일드카드 구문
* `mcp__puppeteer__puppeteer_navigate` — `puppeteer` 서버가 제공하는 `puppeteer_navigate` 도구를 매칭

### Agent (서브에이전트)

`Agent(AgentName)` 규칙으로 Claude가 사용할 수 있는 [서브에이전트](/en/sub-agents)를 제어합니다:

* `Agent(Explore)` — Explore 서브에이전트를 매칭
* `Agent(Plan)` — Plan 서브에이전트를 매칭
* `Agent(my-custom-agent)` — `my-custom-agent`라는 이름의 사용자 정의 서브에이전트를 매칭

이 규칙을 설정의 `deny` 배열에 추가하거나 `--disallowedTools` CLI 플래그를 사용하여 특정 에이전트를 비활성화합니다. Explore 에이전트를 비활성화하려면:

```json  theme={null}
{
  "permissions": {
    "deny": ["Agent(Explore)"]
  }
}
```

## 훅으로 권한 확장

[Claude Code 훅](/en/hooks-guide)은 런타임에 권한 평가를 수행하기 위한 사용자 정의 셸 명령을 등록하는 방법을 제공합니다. Claude Code가 도구를 호출하면 PreToolUse 훅이 권한 프롬프트 전에 실행됩니다. 훅 출력은 도구 호출을 거부하거나, 프롬프트를 강제하거나, 프롬프트를 건너뛰어 호출을 진행시킬 수 있습니다.

프롬프트 건너뛰기는 권한 규칙을 우회하지 않습니다. 훅이 `"allow"`를 반환한 후에도 deny 및 ask 규칙이 평가되므로, 매칭되는 deny 규칙이 여전히 호출을 차단합니다. 이는 [권한 관리](#권한-관리)에서 설명한 deny 우선 원칙을 유지하며, 관리형 설정에서 설정된 deny 규칙도 포함합니다.

차단 훅도 allow 규칙보다 우선합니다. 종료 코드 2로 종료하는 훅은 권한 규칙이 평가되기 전에 도구 호출을 중지하므로, allow 규칙이 호출을 허용했을 경우에도 차단이 적용됩니다. 몇 가지 특정 명령만 차단하면서 모든 Bash 명령을 프롬프트 없이 실행하려면, allow 목록에 `"Bash"`를 추가하고 해당 특정 명령을 거부하는 PreToolUse 훅을 등록하세요. 적용할 수 있는 훅 스크립트는 [보호 파일 편집 차단](/en/hooks-guide#block-edits-to-protected-files)을 참조하세요.

## 작업 디렉토리

기본적으로 Claude는 실행된 디렉토리의 파일에 접근할 수 있습니다. 다음 방법으로 접근을 확장할 수 있습니다:

* **시작 시**: `--add-dir <path>` CLI 인수 사용
* **세션 중**: `/add-dir` 명령 사용
* **영구 설정**: [설정 파일](/en/settings#settings-files)의 `additionalDirectories`에 추가

추가 디렉토리의 파일은 원래 작업 디렉토리와 동일한 권한 규칙을 따릅니다: 프롬프트 없이 읽을 수 있으며, 파일 편집 권한은 현재 권한 모드를 따릅니다.

### 추가 디렉토리는 파일 접근만 부여하며 설정을 적용하지 않음

디렉토리를 추가하면 Claude가 읽고 편집할 수 있는 범위가 확장됩니다. 해당 디렉토리를 전체 설정 루트로 만들지는 않습니다: 대부분의 `.claude/` 설정은 추가 디렉토리에서 검색되지 않지만, 일부 유형은 예외적으로 로드됩니다.

다음 설정 유형은 `--add-dir` 디렉토리에서 로드됩니다:

| 설정 | `--add-dir`에서 로드 여부 |
| :--- | :--- |
| `.claude/skills/`의 [스킬](/en/skills) | 예, 실시간 리로드 지원 |
| `.claude/settings.json`의 플러그인 설정 | `enabledPlugins` 및 `extraKnownMarketplaces`만 |
| [CLAUDE.md](/en/memory) 파일 및 `.claude/rules/` | `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1` 설정 시에만 |

그 외 모든 것(서브에이전트, 명령, 출력 스타일, 훅, 기타 설정)은 현재 작업 디렉토리와 상위 디렉토리, `~/.claude/`의 사용자 디렉토리, 관리형 설정에서만 검색됩니다. 프로젝트 간에 이런 설정을 공유하려면 다음 방법을 사용하세요:

* **사용자 수준 설정**: `~/.claude/agents/`, `~/.claude/output-styles/`, 또는 `~/.claude/settings.json`에 파일을 배치하여 모든 프로젝트에서 사용 가능하게 함
* **플러그인**: 팀이 설치할 수 있는 [플러그인](/en/plugins)으로 설정을 패키징하여 배포
* **설정 디렉토리에서 실행**: 원하는 `.claude/` 설정이 있는 디렉토리에서 Claude Code를 실행

## 권한과 샌드박싱의 상호작용

권한과 [샌드박싱](/en/sandboxing)은 상호 보완적인 보안 계층입니다:

* **권한**은 Claude Code가 사용할 수 있는 도구와 접근할 수 있는 파일 또는 도메인을 제어합니다. 모든 도구(Bash, Read, Edit, WebFetch, MCP 등)에 적용됩니다.
* **샌드박싱**은 Bash 도구의 파일시스템 및 네트워크 접근을 제한하는 OS 수준 강제를 제공합니다. Bash 명령과 그 하위 프로세스에만 적용됩니다.

심층 방어를 위해 둘 다 사용하세요:

* 권한 deny 규칙은 Claude가 제한된 리소스에 접근을 시도하는 것 자체를 차단합니다
* 샌드박스 제한은 프롬프트 인젝션이 Claude의 의사결정을 우회하더라도 Bash 명령이 정의된 경계 밖의 리소스에 도달하는 것을 방지합니다
* 샌드박스의 파일시스템 제한은 별도의 샌드박스 설정이 아닌 Read 및 Edit deny 규칙을 사용합니다
* 네트워크 제한은 WebFetch 권한 규칙과 샌드박스의 `allowedDomains` 목록을 결합합니다

## 관리형 설정

조직에서 Claude Code 설정에 대한 중앙 집중식 제어가 필요한 경우, 관리자는 사용자 또는 프로젝트 설정으로 재정의할 수 없는 관리형 설정을 배포할 수 있습니다. 이러한 정책 설정은 일반 설정 파일과 동일한 형식을 따르며, MDM/OS 수준 정책, 관리형 설정 파일 또는 [서버 관리형 설정](/en/server-managed-settings)을 통해 전달할 수 있습니다. 전달 메커니즘 및 파일 위치는 [설정 파일](/en/settings#settings-files)을 참조하세요.

### 관리형 전용 설정

다음 설정은 관리형 설정에서만 읽힙니다. 사용자 또는 프로젝트 설정 파일에 배치해도 효과가 없습니다.

| 설정 | 설명 |
| :--- | :--- |
| `allowedChannelPlugins` | 메시지를 푸시할 수 있는 채널 플러그인 허용 목록. 설정 시 기본 Anthropic 허용 목록을 대체합니다. `channelsEnabled: true`가 필요합니다. [실행 가능한 채널 플러그인 제한](/en/channels#restrict-which-channel-plugins-can-run)을 참조하세요 |
| `allowManagedHooksOnly` | `true`일 때 사용자, 프로젝트, 플러그인 훅의 로드를 방지합니다. 관리형 훅과 SDK 훅만 허용됩니다 |
| `allowManagedMcpServersOnly` | `true`일 때 관리형 설정의 `allowedMcpServers`만 적용됩니다. `deniedMcpServers`는 여전히 모든 소스에서 병합됩니다. [관리형 MCP 설정](/en/mcp#managed-mcp-configuration)을 참조하세요 |
| `allowManagedPermissionRulesOnly` | `true`일 때 사용자 및 프로젝트 설정에서 `allow`, `ask`, `deny` 권한 규칙을 정의하지 못하게 합니다. 관리형 설정의 규칙만 적용됩니다 |
| `blockedMarketplaces` | 마켓플레이스 소스 차단 목록. 차단된 소스는 다운로드 전에 확인되므로 파일시스템에 접촉하지 않습니다. [관리형 마켓플레이스 제한](/en/plugin-marketplaces#managed-marketplace-restrictions)을 참조하세요 |
| `channelsEnabled` | Team 및 Enterprise 사용자를 위한 [채널](/en/channels) 허용. 미설정이거나 `false`이면 사용자가 `--channels`에 전달하는 내용과 무관하게 채널 메시지 전달을 차단합니다 |
| `pluginTrustMessage` | 설치 전에 표시되는 플러그인 신뢰 경고에 추가되는 사용자 정의 메시지 |
| `sandbox.filesystem.allowManagedReadPathsOnly` | `true`일 때 관리형 설정의 `filesystem.allowRead` 경로만 적용됩니다. `denyRead`는 여전히 모든 소스에서 병합됩니다 |
| `sandbox.network.allowManagedDomainsOnly` | `true`일 때 관리형 설정의 `allowedDomains` 및 `WebFetch(domain:...)` allow 규칙만 적용됩니다. 허용되지 않은 도메인은 사용자에게 프롬프트 표시 없이 자동으로 차단됩니다. 거부된 도메인은 여전히 모든 소스에서 병합됩니다 |
| `strictKnownMarketplaces` | 사용자가 추가할 수 있는 플러그인 마켓플레이스를 제어합니다. [관리형 마켓플레이스 제한](/en/plugin-marketplaces#managed-marketplace-restrictions)을 참조하세요 |

`disableBypassPermissionsMode`는 일반적으로 조직 정책을 강제하기 위해 관리형 설정에 배치되지만, 모든 범위에서 작동합니다. 사용자가 자신의 설정에서 이를 설정하여 bypass 모드에서 스스로를 잠글 수도 있습니다.

<Note>
  [Remote Control](/en/remote-control) 및 [웹 세션](/en/claude-code-on-the-web)에 대한 접근은 관리형 설정 키로 제어되지 않습니다. Team 및 Enterprise 플랜에서 관리자는 [Claude Code 관리자 설정](https://claude.ai/admin-settings/claude-code)에서 이러한 기능을 활성화하거나 비활성화합니다.
</Note>

## Auto 모드 분류기 설정

[Auto 모드](/en/permission-modes#eliminate-prompts-with-auto-mode)는 분류기 모델을 사용하여 각 작업을 프롬프트 없이 실행해도 안전한지 판단합니다. 기본적으로 작업 디렉토리와 현재 리포지토리의 리모트(있는 경우)만 신뢰합니다. 회사의 소스 컨트롤 조직에 푸시하거나 팀 클라우드 버킷에 쓰는 등의 작업은 잠재적 데이터 유출로 차단됩니다. `autoMode` 설정 블록을 사용하여 조직이 신뢰하는 인프라를 분류기에 알려줄 수 있습니다.

분류기는 사용자 설정, `.claude/settings.local.json`, 관리형 설정에서 `autoMode`를 읽습니다. 체크인된 리포지토리가 자체 allow 규칙을 주입할 수 있으므로 `.claude/settings.json`의 공유 프로젝트 설정에서는 읽지 않습니다.

| 범위 | 파일 | 용도 |
| :--- | :--- | :--- |
| 개별 개발자 | `~/.claude/settings.json` | 개인 신뢰 인프라 |
| 하나의 프로젝트, 한 개발자 | `.claude/settings.local.json` | 프로젝트별 신뢰 버킷 또는 서비스, gitignore 대상 |
| 조직 전체 | 관리형 설정 | 모든 개발자에게 강제되는 신뢰 인프라 |

각 범위의 항목은 결합됩니다. 개발자는 개인 항목으로 `environment`, `allow`, `soft_deny`를 확장할 수 있지만, 관리형 설정에서 제공하는 항목은 제거할 수 없습니다. allow 규칙이 분류기 내부에서 차단 규칙의 예외로 작동하므로, 개발자가 추가한 `allow` 항목은 조직의 `soft_deny` 항목을 재정의할 수 있습니다: 결합은 가산적이며 엄격한 정책 경계가 아닙니다. 개발자가 우회할 수 없는 규칙이 필요하면, 분류기가 참조되기 전에 작업을 차단하는 관리형 설정의 `permissions.deny`를 대신 사용하세요.

### 신뢰 인프라 정의

대부분의 조직에서 `autoMode.environment`는 설정해야 할 유일한 필드입니다. 이 필드는 내장 차단 및 허용 규칙을 건드리지 않고 어떤 리포지토리, 버킷, 도메인이 신뢰할 수 있는지 분류기에 알려줍니다. 분류기는 `environment`를 사용하여 "외부"의 의미를 결정합니다: 목록에 없는 대상은 잠재적 유출 대상입니다.

```json  theme={null}
{
  "autoMode": {
    "environment": [
      "Source control: github.example.com/acme-corp and all repos under it",
      "Trusted cloud buckets: s3://acme-build-artifacts, gs://acme-ml-datasets",
      "Trusted internal domains: *.corp.example.com, api.internal.example.com",
      "Key internal services: Jenkins at ci.example.com, Artifactory at artifacts.example.com"
    ]
  }
}
```

항목은 산문이며 정규식이나 도구 패턴이 아닙니다. 분류기는 이를 자연어 규칙으로 읽습니다. 새로운 엔지니어에게 인프라를 설명하듯이 작성하세요. 철저한 environment 섹션은 다음을 다룹니다:

* **조직**: 회사 이름과 Claude Code의 주요 용도(소프트웨어 개발, 인프라 자동화, 데이터 엔지니어링 등)
* **소스 컨트롤**: 개발자가 푸시하는 모든 GitHub, GitLab 또는 Bitbucket 조직
* **클라우드 제공자 및 신뢰 버킷**: Claude가 읽고 쓸 수 있어야 하는 버킷 이름 또는 접두사
* **신뢰 내부 도메인**: `*.internal.example.com` 같은 API, 대시보드, 네트워크 내 서비스의 호스트이름
* **핵심 내부 서비스**: CI, 아티팩트 레지스트리, 내부 패키지 인덱스, 인시던트 도구
* **추가 컨텍스트**: 분류기가 위험으로 취급해야 하는 규제 산업 제약, 멀티 테넌트 인프라 또는 컴플라이언스 요구사항

유용한 시작 템플릿: 대괄호 안의 필드를 채우고 해당되지 않는 줄은 제거하세요:

```json  theme={null}
{
  "autoMode": {
    "environment": [
      "Organization: {COMPANY_NAME}. Primary use: {PRIMARY_USE_CASE, e.g. software development, infrastructure automation}",
      "Source control: {SOURCE_CONTROL, e.g. GitHub org github.example.com/acme-corp}",
      "Cloud provider(s): {CLOUD_PROVIDERS, e.g. AWS, GCP, Azure}",
      "Trusted cloud buckets: {TRUSTED_BUCKETS, e.g. s3://acme-builds, gs://acme-datasets}",
      "Trusted internal domains: {TRUSTED_DOMAINS, e.g. *.internal.example.com, api.example.com}",
      "Key internal services: {SERVICES, e.g. Jenkins at ci.example.com, Artifactory at artifacts.example.com}",
      "Additional context: {EXTRA, e.g. regulated industry, multi-tenant infrastructure, compliance requirements}"
    ]
  }
}
```

구체적인 컨텍스트를 제공할수록 분류기가 일상적인 내부 작업과 유출 시도를 더 잘 구분할 수 있습니다.

모든 것을 한 번에 채울 필요는 없습니다. 합리적인 단계별 적용: 기본값으로 시작하고 소스 컨트롤 조직과 핵심 내부 서비스를 추가하면, 자체 리포지토리에 푸시하는 것 같은 가장 흔한 오탐을 해결할 수 있습니다. 그다음 신뢰 도메인과 클라우드 버킷을 추가합니다. 나머지는 차단이 발생할 때 채우세요.

### 차단 및 허용 규칙 재정의

두 가지 추가 필드로 분류기의 내장 규칙 목록을 대체할 수 있습니다: `autoMode.soft_deny`는 차단 대상을 제어하고, `autoMode.allow`는 적용할 예외를 제어합니다. 각각은 자연어 규칙으로 읽히는 산문 설명의 배열입니다.

분류기 내부에서 우선순위는: `soft_deny` 규칙이 먼저 차단한 다음 `allow` 규칙이 예외로 재정의하고, 명시적 사용자 의도가 둘 다를 재정의합니다. 사용자의 메시지가 Claude가 수행하려는 정확한 작업을 직접적이고 구체적으로 설명하면, `soft_deny` 규칙이 매칭되더라도 분류기가 이를 허용합니다. 일반적인 요청은 해당되지 않습니다: Claude에게 "리포를 정리해줘"라고 요청하는 것은 강제 푸시를 승인하지 않지만, "이 브랜치를 강제 푸시해줘"라고 요청하는 것은 승인합니다.

완화하려면: 기본값이 PR 리뷰, CI 또는 스테이징 환경으로 이미 보호하고 있는 항목을 차단할 때 `soft_deny`에서 규칙을 제거하거나, 분류기가 기본 예외에서 다루지 않는 일상적인 패턴을 반복적으로 플래깅할 때 `allow`에 추가합니다. 강화하려면: 기본값이 놓치는 환경별 위험에 대해 `soft_deny`에 추가하거나, 기본 예외를 차단 규칙에 적용하기 위해 `allow`에서 제거합니다. 모든 경우에 `claude auto-mode defaults`를 실행하여 전체 기본 목록을 가져온 다음 복사하고 편집하세요: 빈 목록에서 시작하지 마세요.

```json  theme={null}
{
  "autoMode": {
    "environment": [
      "Source control: github.example.com/acme-corp and all repos under it"
    ],
    "allow": [
      "Deploying to the staging namespace is allowed: staging is isolated from production and resets nightly",
      "Writing to s3://acme-scratch/ is allowed: ephemeral bucket with a 7-day lifecycle policy"
    ],
    "soft_deny": [
      "Never run database migrations outside the migrations CLI, even against dev databases",
      "Never modify files under infra/terraform/prod/: production infrastructure changes go through the review workflow",
      "...copy full default soft_deny list here first, then add your rules..."
    ]
  }
}
```

<Danger>
  `allow` 또는 `soft_deny`를 설정하면 해당 섹션의 전체 기본 목록이 대체됩니다. `soft_deny`에 단일 항목만 설정하면, 모든 내장 차단 규칙(강제 푸시, 데이터 유출, `curl | bash`, 프로덕션 배포 등)이 폐기되어 허용됩니다. 안전하게 커스터마이즈하려면 `claude auto-mode defaults`를 실행하여 내장 규칙을 출력하고, 설정 파일에 복사한 다음, 자체 파이프라인과 위험 허용 범위에 맞게 각 규칙을 검토하세요. 인프라가 이미 완화하고 있는 위험에 대해서만 규칙을 제거하세요.
</Danger>

세 섹션은 독립적으로 평가되므로, `environment`만 설정하면 기본 `allow` 및 `soft_deny` 목록은 그대로 유지됩니다.

### 기본값 및 유효 설정 확인

`allow` 또는 `soft_deny`를 설정하면 기본값이 대체되므로, 커스터마이즈는 전체 기본 목록을 복사하는 것부터 시작하세요. 세 가지 CLI 하위 명령이 확인 및 검증에 도움됩니다:

```bash  theme={null}
claude auto-mode defaults  # 내장 environment, allow, soft_deny 규칙
claude auto-mode config    # 분류기가 실제로 사용하는 것: 설정이 있으면 해당 설정, 없으면 기본값
claude auto-mode critique  # 사용자 정의 allow 및 soft_deny 규칙에 대한 AI 피드백
```

`claude auto-mode defaults`의 출력을 파일에 저장하고, 정책에 맞게 목록을 편집한 후, 결과를 설정 파일에 붙여넣으세요. 저장 후 `claude auto-mode config`를 실행하여 유효 규칙이 예상과 일치하는지 확인하세요. 사용자 정의 규칙을 작성했다면 `claude auto-mode critique`가 이를 검토하고 모호하거나 중복되거나 오탐을 유발할 수 있는 항목을 표시합니다.

## 설정 우선순위

권한 규칙은 다른 모든 Claude Code 설정과 동일한 [설정 우선순위](/en/settings#settings-precedence)를 따릅니다:

1. **관리형 설정**: 명령줄 인수를 포함한 다른 어떤 수준에서도 재정의할 수 없음
2. **명령줄 인수**: 임시 세션 재정의
3. **로컬 프로젝트 설정** (`.claude/settings.local.json`)
4. **공유 프로젝트 설정** (`.claude/settings.json`)
5. **사용자 설정** (`~/.claude/settings.json`)

도구가 어떤 수준에서든 거부되면, 다른 수준에서 이를 허용할 수 없습니다. 예를 들어, 관리형 설정의 deny는 `--allowedTools`로 재정의할 수 없으며, `--disallowedTools`는 관리형 설정이 정의한 것 이상의 제한을 추가할 수 있습니다.

사용자 설정에서 권한이 허용되었지만 프로젝트 설정에서 거부된 경우, 프로젝트 설정이 우선하여 권한이 차단됩니다.
